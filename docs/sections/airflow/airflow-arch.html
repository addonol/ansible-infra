Orchestration Map

System Architecture

A decoupled grid designed for high availability and secure communication, leveraging a zero-trust identity layer.

Click to enlarge diagram

Logical Segregation

The infrastructure is partitioned into four isolated zones—Security, Core, Persistence, and External—to prevent lateral movement and contain potential failures.

Zero-Trust Ingress

All traffic is intercepted at the edge by an Identity-Aware Proxy, offloading authentication from the core services and enforcing strict access control.

M2M Security

Automated JWT & Refresh Token lifecycles secure the Machine-to-Machine handshakes between workers and external APIs, shielding metadata from exposure.

The Operational Lifecycle: Step-by-Step

The following narrative describes the complete journey of an orchestration task, tracing the flow from the initial user request to the final external API execution.

Access & Identity Validation (Steps 1-3): The workflow begins at the network edge. When a user attempts to access the Airflow UI, the cas-proxy intercepts the request to verify session validity. Unauthenticated traffic is redirected to the cas-server (or Mock) for credential validation. Upon success, a secure service ticket is issued, granting a narrow-scope entry into the internal control plane.
Ingress & State Visualization (Steps 4-5): Once through the perimeter, the api-server (FastAPI) acts as the stateless bridge for the ui-react frontend. Instead of the UI querying the database directly, it interacts solely with RESTful endpoints. This centralizes the execution logic and ensures that any action taken in the browser is validated by the server before reaching the persistence layer.
Metadata Sync & Scheduling (Steps 6-7): While the api-server maintains the real-time state with postgres, the scheduler operates in parallel to identify tasks ready for execution. It relies on the dag-processor to receive serialized JSON versions of the Python DAG files, ensuring that complex parsing logic never interferes with the critical scheduling heartbeat.
Task Queueing & Consumption (Steps 8-9): When a task dependency is met, the Scheduler pushes a command into the rabbitmnq broker. This durable message queue holds the task definition until an available celery-worker, running in a secure rootless container, claims the message. This decoupling allows the system to scale workers independently of the control plane load.
External Handshake & Execution (Step 10): The Worker executes the Python code within its isolated namespace. For tasks requiring external data, the worker initiates a secure M2M handshake to retrieve an Access JWT. For long-running operations, it utilizes a Refresh Token to maintain authorized communication without requiring manual intervention or hardcoded secrets.
Completion & Async Resumption (Step 11): Upon task completion, the Worker sends a final status update back to the api-server via a JWT-signed request. In cases where the task was deferred to save resources, the triggerer monitors the event loop and signals the server when the external condition is met, allowing the workflow to resume seamlessly.
