Identity Framework

Authentication Strategies

The Forge Framework supports a multi-layered authentication strategy, choosing the optimal mechanism based on the actor (Human vs. Machine) and the required trust level.

Static Access

1. API Keys (Service-to-Service)

The API Key is a persistent, long-lived token passed in HTTP headers. In this framework, it is primarily used for simple identification between legacy internal services or third-party providers that do not support dynamic token exchange.

Security Posture: While easy to implement, API keys carry higher risk as they are static. If a key is compromised, access remains open until manual revocation. They provide identity but lack the rich, time-bound "claims" found in modern tokens.

Stateless Claims

2. JWT (Stateless Metadata)

The JSON Web Token (JWT) is a cryptographically signed standard (RFC 7519) used for secure, stateless communication. Our API-Server utilizes JWTs to verify requests without hitting the metadata database every time.

Technical Edge: Because the token contains encrypted claims (user roles, expiration, permissions), the server simply validates the signature. This decoupling is essential for the high-performance requirements of Airflow 3 execution cycles.

SSO Standard

3. OIDC (OpenID Connect Identity Layer)

OpenID Connect (OIDC) sits on top of the OAuth 2.0 framework to provide a complete identity protocol. Unlike an API Key (an identifier) or a JWT (a format), OIDC defines the entire handshake flow between the User, the Client (Airflow), and the Identity Provider (CAS Server).

In this infrastructure, OIDC allows Airflow to delegate authentication to the CAS Server. The user logs in once at the perimeter, and the CAS issues an ID Token that proves the human identity behind the request, enabling seamless Single Sign-On (SSO).

Pros: Centralized Identity

User management is centralized. Disabling a user in the CAS Server immediately revokes their access across all connected framework components.

Cons: Handshake Complexity

Requires a stable network connection between the Proxy and the CAS Server. A failure in the OIDC provider locks out all human operators from the UI.

Architectural Recommendation

The framework enforces a Hybrid Model: We use OIDC/CAS for human administrators accessing the UI, while JWTs and API Keys (stored in Ansible Vault) secure the high-speed Machine-to-Machine (M2M) traffic between Workers, the API-Server, and external datasets.
