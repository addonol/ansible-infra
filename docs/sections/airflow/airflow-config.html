Jinja2 Automation

Configuration Deep-Dive

The airflow.cfg.j2 template is the heart of the deployment. It dynamically adapts to Airflow 3 standards while maintaining backward compatibility.

Core & Auth Management

In the transition to Airflow 3, the selection of the auth_manager has become a critical architectural decision. The framework utilizes conditional Jinja2 blocks to automatically toggle between the legacy FAB (Flask App Builder) manager and the modern FastAPI SimpleAuthManager. When the airflow_use_fastapi flag is enabled, the system bypasses traditional RBAC in favor of a high-performance identity provider. This allows for a truly stateless bootstrap, as admin users are provisioned directly via the configuration file, eliminating the need for manual user creation during the initial cluster setup.

The airflow.cfg.j2 Template

[core]
executor = {{ airflow_common_executor }}
dags_folder = /opt/airflow/dags
fernet_key = {{ airflow_common_fernet_key }}
internal_api_secret_key = {{ airflow_common_jwt_secret }}

# Dynamic Auth Manager Selection
{% if airflow_use_fastapi | default(false) %}
auth_manager = airflow.api_fastapi.auth.managers.simple.simple_auth_manager.SimpleAuthManager
{% else %}
auth_manager = airflow.providers.fab.auth_manager.fab_auth_manager.FabAuthManager
{% endif %}

[database]
{% set postgres_uri = "postgresql+psycopg2://" ~ airflow_common_postgres_db_user ~ ":" ~ (airflow_common_postgres_db_pass | urlencode) %}
sql_alchemy_conn = {{ postgres_uri }}@{{ airflow_common_postgres_instance }}:5432/{{ airflow_common_postgres_db_name }}

[celery]
{% set rabbitmq_uri = "amqp://" ~ airflow_common_rabbitmq_user ~ ":" ~ (airflow_common_rabbitmq_pass | urlencode) %}
broker_url = {{ rabbitmq_uri }}@{{ airflow_common_rabbitmq_instance }}:{{ airflow_common_rabbitmq_management_port }}//
result_backend = db+postgresql://{{ airflow_common_postgres_db_user }}:{{ airflow_common_postgres_db_pass | urlencode }}@{{ airflow_common_postgres_instance }}:{{ airflow_common_postgres_port}}/{{ airflow_common_postgres_db_name }}

[api_fastapi]
enabled = {{ airflow_use_fastapi | default(false) | lower }}
jwt_secret = {{ airflow_common_jwt_secret }}

Connection String Sanitization

The construction of connection strings (URIs) for PostgreSQL and RabbitMQ is a delicate process due to potential special characters in passwords. To prevent deployment-breaking syntax errors, the framework enforces the urlencode filter on all credentials. This ensures that the result_backend and broker_url are always RFC-compliant, allowing Celery workers to maintain stable persistence cycles with the metadata store.

Encryption & JWT Security

Security is baked directly into the configuration layer through a multi-key strategy. The Fernet Key is utilized for encrypting connection secrets within the database, while a shared JWT Secret is distributed across the API-Server, Webserver, and Workers. This shared secret allows for seamless M2M (Machine-to-Machine) validation, ensuring that every internal request is cryptographically signed and verified before execution.

Ansible Vault & Variable Management

Sensitive data—Database passwords, fernet_key, and jwt_secret—are encrypted using Ansible Vault. Decryption occurs only during deployment.

Automated Deployment Lifecycle

Infrastructure & Secret Provisioning: The role first establishes a dedicated Podman Network for secure inter-container routing. It then provisions the airflow.cfg as a secure Podman Secret, guaranteeing that sensitive parameters are never written to the host's persistent disk in unencrypted form.
Database & Identity Migration: Before the main services launch, a transient "init" container is triggered to execute airflow db migrate. This process conditionally initializes the modern FastAPI SimpleAuthManager or legacy RBAC users, then self-destructs (rm: true) to minimize the environment's attack surface.
Volume & Permission Management: Host directories for DAGs, Logs, and Plugins are provisioned with specific 0775 permissions. By utilizing userns: keep-id mapping, the role allows rootless containers to write to these volumes as the host user, resolving traditional "Permission Denied" issues in rootless environments.
Orchestrated Service Launch: The final stage maps Airflow commands (Webserver, Scheduler, Worker) to individual containers. Each container is launched with strictly limited resources and mounts only the necessary secrets, ensuring a fully decoupled and resilient execution tier.
