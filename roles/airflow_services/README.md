# ⚙️ Ansible Role: Airflow Services (Multi-Purpose)

This Ansible role is a dynamic orchestrator for **Airflow 3** components. It is designed to deploy any Airflow service (**Webserver**, **Scheduler**, **Worker**, or **Triggerer**) using a single, unified logic with **Podman** in **rootless mode**.

## 🌟 Key Features
*   **Unified Service Engine**: One role to rule them all. Just pass the `airflow_service_name` to deploy the specific component.
*   **Airflow 3 Native**: Fully compatible with Airflow 3.x execution commands and internal API structures.
*   **Configuration Inheritance**: Automatically reuses the `airflow_config_secret` generated by the `airflow_common` role.
*   **Smart Port/Health Management**: Conditionally maps ports (8080) and HTTP healthchecks only when deploying the `webserver` component.
*   **Instance Isolation**: Supports `inventory_hostname` suffixing to allow parallel testing via Molecule without naming conflicts.

## 🛠 Prerequisites
*   **Core Setup**: Must be run after `airflow_common` (for secrets and DB migrations).
*   **Infrastructure**: Requires functional PostgreSQL and RabbitMQ instances.
*   **Target Host**: Podman installed (rootless mode).

## 📦 Role Variables

### Dynamic Service Selection (`defaults/main.yml`)



| Variable | Default | Description |
| :--- | :--- | :--- |
| `airflow_services_name` | `webserver` | Service to launch (`webserver`, `scheduler`, `worker`, `triggerer`). |
| `airflow_image` | `apache/airflow:3.0.0` | Official Airflow 3 image. |
| `airflow_webserver_port` | `8080` | Host port for the Web UI. |
| `airflow_dags_folder` | `/path/to/dags` | Host path for DAGs persistence. |

### Configuration Mapping (From Common)



| Variable | Default | Description |
| :--- | :--- | :--- |
| `airflow_config_secret_name` | `airflow_config_secret` | Name of the secret containing `airflow.cfg`. |
| `shared_network` | `infra_net` | Podman network for inter-container communication. |

## 🚀 Deployment Example

The best way to use this role is within a loop in your main playbook:

```yaml
- name: Launch Airflow 3 Services
  include_role:
    name: airflow_services
  vars:
    airflow_service_name: "{{ item }}"
  loop:
    - webserver
    - scheduler
    - triggerer
    - worker
```

## 🧪 Verification & Health

When deploying the webserver, the role automatically includes a native Podman healthcheck:

* **Command**: curl -f http://localhost:8080/health
* **Wait Period**: 30s start period to allow Airflow 3 internal initialization.
